# LCD12864 模块说明

## 这个模块是做什么的

这是一个适用于 128×64 像素单色点阵 LCD 屏幕的驱动模块。驱动代码以 UC1701X 控制器为示例原型（市面上很多 12864 屏幕——比如用 ST7565、ST7567、SSD1306 的——控制逻辑都类似），提供了一个从"我想在屏幕上画点什么"到"像素真的亮了"的完整通路。

模块按照**驱动核心**和**平台端口**两层来组织，核心层只管"怎么控制屏幕"，不管"怎么操作 MCU 的 SPI 外设"。所以如果将来要换一块单片机，只用改写端口层，核心层可以原封不动搬过去。

## 架构分层

驱动内部可以看作三个层次叠在一起，从上到下分别是**配置层**、**协议层**和**硬件适配层**。下面慢慢说每一层做了什么。

### 配置层

用户在使用屏幕之前，需要先回答一个问题：屏幕插在哪些引脚上，SPI 跑多快？

这个答案封装在 `lcd12864_config_t` 这个结构体里。它包含五个引脚号（SCLK、SDA、RS/DC、CS、RESET）和一个 SPI 时钟频率。用户在创建屏幕实例的时候把这个结构体传进来，驱动就知道怎么跟硬件打交道了。这样做的好处很明显：同一份代码编译出来，换一组引脚就能适配不同的硬件连接，不用改驱动源码。

配置层的另一部分是用 Kconfig 写的菜单。如果用户不想在代码里手写引脚号，也可以在 `menuconfig` 的图形界面里选好，然后通过 `CONFIG_LCD12864_PIN_xxx` 这些宏来引用。Kconfig 提供了一套默认值（SCLK=1, SDA=2, RS=3, RESET=4, CS=5, SPI 1MHz），方便快速上手。

### 协议层（驱动核心 `lcd12864.c`）

协议层是驱动的大脑，它知道屏幕控制器应该按什么顺序接受命令和数据进行正确显示。这片大脑做的事包括：

- **硬件初始化流程**：给屏幕发送一组预定义的命令序列（比如 0xE2 软件复位、0x2F 打开电源模块、0xAF 开显示），让控制器从出厂状态进入可写入显示数据的工作状态。
- **帧缓冲区管理**：在内存中维护一块 1024 字节的缓冲区，对应屏幕的 128×64 个像素。所有"画点什么"的操作——画点、画矩形、写字符——都先修改这块缓冲区，最后统一刷到硬件上。这种设计有个好处：用户画一个很复杂的画面时，屏幕不会因为逐笔刷新而闪烁。
- **坐标换算**：UC1701X 这类控制器用的是"页寻址"模式——把 64 行分成 8 个页，每页 8 行。驱动内部要把用户传来的像素坐标换算成页号加上该页内的偏移量。
- **字符渲染**：配合 `fonts/font_8x16.h` 字库，把 ASCII 字符的点阵数据渲染到帧缓冲区。字库是水平行排列的，而硬件需要垂直列排列，所以中间要做一次 8×8 的位转置（bit transpose）。
- **并发保护**：用 FreeRTOS 互斥锁保护帧缓冲区和 SPI 传输，防止两个任务同时操作屏幕互相干扰。

协议层完全不关心 SPI 是怎么发出去的、GPIO 是怎么翻转的。它调用端口层的函数指针来完成这些操作，所以端口层换掉，协议层不用动。

### 硬件适配层（端口层 `port/esp_idf_spi.c`）

端口层是驱动的手和脚，它负责真正物理层面的信号操作。在 ESP-IDF 上，它做下面这些事情：

- **初始化 SPI 总线**：调用 `spi_bus_initialize` 配置 SPI2 主机的 SCLK、MOSI 引脚，然后调用 `spi_bus_add_device` 把屏幕挂载为一个 SPI 设备。
- **手动控制片选**：因为每次传输命令或数据时都需要独立拉低/拉高 CS 引脚，所以不使用硬件自动片选，而是通过 GPIO 手动控制。
- **RS 引脚切换**：将 RS（也叫 DC/A0）引脚置低时，SPI 上发的是命令；置高时，发的是数据。
- **SPI 帧传输**：每次事务发送 8 位数据，用 `spi_device_polling_transmit` 以轮询模式完成，不依赖 DMA 中断。
- **批量数据发送**：`write_data_bulk` 在一次 SPI 事务中发送连续的多字节数据，显著减少事务开销（比如整页 128 字节一次送完）。
- **硬件复位时序**：拉低 RESET 引脚 100ms 再拉高，让控制器进入已知状态。

端口层与具体 MCU 的耦合方式是通过 `lcd12864_port_ops_t` 这个**函数指针表**。它是一个结构体，里面每个字段都是一个函数指针：`init`、`reset`、`delay_ms`、`write_cmd`、`write_data`、`write_data_bulk`、`deinit`。驱动核心在运行时获取这个表的副本，然后通过它调用具体的实现函数。不同的平台只需要提供不同的 `lcd12864_port_ops_t` 实例，核心层就能无缝工作。

## 实现原理详解

### 初始化流程

屏幕从上电到正常显示，中间经历了一条完整的初始化链路（在 `lcd12864_create` 内部自动完成）：

1. **分配上下文**：`calloc` 分配一块内存存放屏幕实例的所有状态，包括操作表副本、光标坐标、显示开关状态等。分配完就把所有字段清零，避免未初始化值。
2. **获取操作表**：把全局常量 `lcd12864_port_ops` 的值复制到上下文内。这样即使外层改了全局表，已有屏幕实例不受影响。
3. **初始化硬件**：调用端口层的 `init` 函数，配置 SPI 总线和 GPIO 引脚。
4. **硬件复位**：RESET 拉低 100ms → 拉高 100ms，等待控制器内部稳定。
5. **分配帧缓冲区**：`malloc(1024)`，存放完整的屏幕显存镜像。
6. **创建互斥锁**：`xSemaphoreCreateMutex()`，后续所有绘图操作都需要先获取这个锁。
7. **软件初始化序列**：依次发送命令字节，配置控制器的内部寄存器。
8. **清屏并刷新**：帧缓冲区全部置零，然后调用 `flush` 送一次空画面到硬件。

如果以上任何一个步骤失败，已经分配的资源会全部回滚释放，避免内存泄漏。

软件初始化序列发送的命令包括：

| 命令字节            | 作用                               |
| --------------- | -------------------------------- |
| `0xE2`          | 软件复位，让控制器内部的寄存器回到默认值             |
| `0x2F`          | 打开内部的升压器、电压调节器和电压跟随器，产生屏幕需要的工作电压 |
| `0xAF`          | 打开显示输出，屏幕开始刷新                    |
| `0xA0` / `0xC8` | 设置 SEG 和 COM 的扫描方向，修正镜像          |

这些命令是针对 UC1701X 的。如果实际使用的屏幕控制器不同，命令序列需要按对应数据手册调整。

### 帧缓冲区与刷屏机制

帧缓冲区是一块 1024 字节的堆内存，布局方式与 UC1701X 的显示 RAM 完全一致：每 128 字节对应一页（水平一排 128 列 × 垂直 8 行 = 1024 bit = 128 字节），共 8 页。

屏幕上 128 个宽、64 个高的坐标系中，任意一个像素点的位置由它在哪个页（`y / 8`）、哪一列（`x`）以及在该页字节中的哪个位（`y % 8`）决定。

用户调用 `flush` 时，驱动按页遍历，对每一页做三件事：

1. 发送页地址命令（`0xB0 | page`），告诉控制器将 RAM 指针移动到该页。
2. 发送列地址命令（`0x10 | 0` 和 `0x00 | 0`），将列指针指向第 0 列。
3. 通过 `write_data_bulk` 一次传输 128 字节，将该页内容送入显示 RAM。

这种逐页刷新的方式虽然比单次 1024 字节传输开销稍大，但兼容性好——大多数页寻址控制器都支持这个流程。

### ASCII 字符渲染与位转置

字库 `font_8x16.h` 以 C 语言静态数组的形式存放了 95 个 ASCII 可打印字符（0x20 空格到 0x7E 波浪线）的点阵数据，每个字符 16 字节。

字库的组织方式是**水平行排列**：每个字节对应字符中的一行 8 个水平像素（bit7 为最左、bit0 为最右）。前 8 字节是字符的上半部分，后 8 字节是下半部分。

而 UC1701X 的显示 RAM 是**垂直列排列**：每个字节对应一列中从上到下的 8 个像素（bit0 为最上、bit7 为最下）。

这两个排列方向恰好差了 90 度。所以不能直接把字库数据拷过去。驱动中的 `write_char` 和 `putc` 函数在写帧缓冲区之前，对字库数据做了一次 8×8 的位转置——可以理解成把字符点阵顺时针旋转 90 度，再逐列填入帧缓冲区。

### 并发安全机制

在多任务系统（ESP-IDF 基于 FreeRTOS）里，如果有两个任务同时操作同一个屏幕——比如一个任务正在画菜单，另一个任务在画进度条——就可能出现帧缓冲区数据错乱、SPI 事务交叉等异常。

解决方法是每个屏幕实例持有一个 FreeRTOS 互斥锁（递归互斥语义）。用户在绘制一段完整的画面之前调用 `lcd12864_lock`，绘制完成（含 `flush`）后调用 `lcd12864_unlock`。如果锁已被其他任务持有，当前任务会阻塞等待，直到锁被释放。

互斥锁是在 `create` 阶段创建的，在 `deinit` 阶段通过 `vSemaphoreDelete` 销毁。

## 对外接口

公开 API 分为几个功能组：

### 生命周期管理

| 函数                        | 功能                      | 返回                 |
| ------------------------- | ----------------------- | ------------------ |
| `lcd12864_create(cfg)`    | 创建屏幕实例，自动完成全部初始化流程      | `void*`（失败返回 NULL） |
| `lcd12864_init(handle)`   | 重新执行软件初始化序列             | `esp_err_t`        |
| `lcd12864_deinit(handle)` | 释放屏幕资源，关闭显示、释放 SPI、回收内存 | 无                  |

`create` 是主要的入口。`init` 在 `create` 内已被调用过一次，用户通常不需要再主动调它——除非有特殊需求（比如系统从深度休眠中恢复，需要重新配置屏幕）。

### 显示控制

| 函数                             | 功能                  |
| ------------------------------ | ------------------- |
| `lcd12864_display_on(handle)`  | 恢复显示，屏幕内容保留         |
| `lcd12864_display_off(handle)` | 关闭显示，屏幕内容保留（只是不再刷新） |
| `lcd12864_clear(handle)`       | 帧缓冲全部清零，光标回左上角      |
| `lcd12864_flush(handle)`       | 将帧缓冲区内容整屏刷到硬件       |

`clear` 只清除内存中的帧缓冲区，不会立刻影响屏幕实际显示——要等调了 `flush` 才会生效。`display_off` 不会破坏帧缓冲区数据，重新开显示后内容还在。

### 基于光标的文本输出

| 函数                                   | 功能                          |
| ------------------------------------ | --------------------------- |
| `lcd12864_set_cursor(handle, x, y)`  | 设置光标位置（x: 列像素，y: 页号）        |
| `lcd12864_write_char(handle, ch)`    | 在当前光标位置渲染一个 ASCII 字符，光标自动右移 |
| `lcd12864_write_string(handle, str)` | 从光标位置开始渲染整个字符串              |

光标是一对内部坐标（像素列、页号），由 `set_cursor` 设置，由 `write_char` 自动推进（每个字符右移 8 列，超出 128 列则换行）。

### 基于帧缓冲的绘图 API

| 函数                                              | 功能                                   |
| ----------------------------------------------- | ------------------------------------ |
| `lcd12864_draw_pixel(handle, x, y, color)`      | 在像素坐标 (x, y) 画一个点（x: 0~127, y: 0~63） |
| `lcd12864_fill_rect(handle, x, y, w, h, color)` | 填充矩形区域                               |
| `lcd12864_putc(handle, x, y, c)`                | 在指定像素坐标渲染一个 ASCII 字符                 |
| `lcd12864_puts(handle, x, y, str)`              | 在指定位置开始渲染字符串                         |

这些函数**只修改帧缓冲区**，不直接影响硬件。画完所有内容后，需要调一次 `flush` 让屏幕显示出结果。这样做的好处是避免屏幕闪烁，也让多步绘图操作有更好的原子性。

`draw_pixel` 和 `fill_rect` 的 `color` 参数是布尔值——`true` 表示点亮（白色），`false` 表示熄灭（黑色）。

### 并发保护

| 函数                        | 功能            |
| ------------------------- | ------------- |
| `lcd12864_lock(handle)`   | 获取互斥锁（可能阻塞等待） |
| `lcd12864_unlock(handle)` | 释放互斥锁         |

典型的多任务用法：

```c
lcd12864_lock(lcd);
lcd12864_clear(lcd);
lcd12864_puts(lcd, 0, 0, "Hello World");
lcd12864_puts(lcd, 0, 2, "ESP32 LCD Demo");
lcd12864_flush(lcd);
lcd12864_unlock(lcd);
```

## 底层硬件控制原理

### SPI 通信时序

屏幕通过一种类 SPI 的串行协议与单片机通信，涉及以下信号线：

- **SCLK**：串行时钟，由单片机产生。每个有效时钟沿推动一位数据传输。本模块配置为 SPI mode 0（空闲时时钟为低，上升沿采样数据）。
- **SDA / MOSI**：数据线。命令字节和数据字节都从这里串行送出。
- **RS / DC**：这条线决定了 SPI 上发的字节是命令还是数据。低电平表示当前发送的是命令（比如"设置页地址"），高电平表示当前发送的是数据（比如"显示 RAM 的内容"）。
- **CS**：片选，低电平有效。拉低后一次 SPI 传输开始，拉高后传输结束。
- **RESET**：复位，低电平有效。拉低至少 100µs 会触发控制器进入复位状态。拉高后等待 100ms，控制器恢复运行。

### 页寻址模式

UC1701X 把 64 行像素均匀分成 8 个页（Page 0 到 Page 7），每个页包含 8 行。往显示 RAM 写入一个字节时，这个字节的 8 个 bit 分别对应所在页内从上到下的 8 个像素。

写入之前，需要先告诉控制器要写哪个页（通过 0xB0 | page 命令）、从哪一列开始写（分两次发送列地址的高 4 位和低 4 位）。之后每写入一个字节，列地址自动加 1。

整屏刷新就是利用这个"按页随机寻址 + 列自动递增"的机制：每页设定一次起始地址，然后连续写入 128 字节完成整页填充。

### SPI 协议与 lcd12864 端口层的异同

对比 LCD12864 的端口函数和 LoRa 模块的端口函数，可以看到这两类设备的 SPI 通信方式不同：

- **LCD12864**：需要 RS 引脚来区分命令和数据。写命令时 RS=0，写数据时 RS=1。片选在每帧开始时拉低、结束后拉高。
- **LoRa**：不需要 RS 引脚。通过地址字节的最高位（bit7）来区分是读还是写。bit7=1 表示写寄存器，bit7=0 表示读寄存器。

但底层 SPI 传输的实现方式是一样的：都用 `spi_device_polling_transmit` 发送 8 位帧，都用手动 GPIO 控制片选。所以两个模块的端口实现风格非常接近，可以互相参照。

## 对 RTOS 的兼容性

这个模块在设计上考虑了 FreeRTOS 环境：

- **互斥锁**：每个屏幕实例持有一个 FreeRTOS 互斥锁（`SemaphoreHandle_t`），调用 `lock` 时用 `xSemaphoreTake(portMAX_DELAY)` 阻塞等待。这意味着可以在多个任务中安全地操作同一个屏幕。
- **非阻塞延时**：端口层使用 `vTaskDelay(pdMS_TO_TICKS(ms))` 实现延时，当前任务在等待期间会让出 CPU 给其他任务，而不是忙等浪费 CPU 周期。
- **不依赖中断**：所有 SPI 传输都是轮询模式（`spi_device_polling_transmit`），不占用中断资源，也不会受到中断优先级反转的干扰。
- **任务栈考虑**：`esp_port_init` 中在栈上构造了几个结构体（如 `spi_bus_config_t`、`spi_device_interface_config_t`），每个几十字节，在默认任务栈大小下没问题。但如果任务栈被压得非常小（比如 < 2048 字节），可能需要加大。

在不使用 RTOS 的裸机环境下，互斥锁函数变成空操作即可，但需要找一个替代的延时函数（通常是硬件定时器忙等循环）。

## 硬件资源占用

### 堆内存

- **上下文结构体**：47 字节（包含操作表 7 个函数指针 56 字节 + 若干变量，实际因对齐可能略多），在 `create` 时从堆上分配。
- **帧缓冲区**：1024 字节，在 `create` 时独立分配。
- **互斥锁**：FreeRTOS 互斥锁占用一小段内存，在内核内部管理。

合计约 **1.1 KB** 堆内存。对于 ESP32 的 520KB SRAM 来说非常宽松。

### SPI 资源

- 使用 SPI2 主机（具体是 VSPI 还是 HSPI 由芯片型号决定）。
- 未使用 MISO 引脚（屏幕只需要写数据，不需要回读）。
- 使用 DMA 通道自动分配（`SPI_DMA_CH_AUTO`）。如果系统 DMA 资源紧张，可以改为 `SPI_DMA_DISABLED`。
- 同时只能有一个任务在总线上通信（由 `lock`/`unlock` 保证）。

### GPIO 资源

占用 5 个 GPIO：SCLK、MOSI、RS、CS、RESET。如果系统中已经有其他 SPI 设备共享同一总线，SCLK 和 MOSI 可以复用，但 CS 引脚是屏幕独占的。

### CPU 占用

- `flush` 操作在 4MHz SPI 时钟下发送约 1050 字节（8 页 × 128 字节 + 24 个命令字节），耗时约 2.1ms。在这段时间内当前任务被 SPI 轮询阻塞，不会让出 CPU。
- 其他绘图操作（`draw_pixel`、`putc` 等）只操作内存，时间在微秒级。

## 与其他模块的兼容性

### 共享 SPI 总线

如果系统中有多个设备共用 SPI2 总线（比如 LCD12864 和 SD 卡），需要注意：

- 两个 CS 引脚必须是不同的 GPIO。
- 在 `init` 中第一次初始化 SPI 总线后，第二次再 `spi_bus_initialize` 同一个主机会返回错误。方案有两种：
  - 方案一：由上层代码负责一次性初始化 SPI 总线，各个驱动只调用 `spi_bus_add_device`。
  - 方案二：在驱动中判断总线是否已经初始化，跳过重复初始化（当前版本未实现此逻辑，直接使用 `ESP_ERROR_CHECK`，初始化失败时会触发断言）。

### 与 LoRa 模块的共存

如果系统同时使用 LCD12864 和 LoRa 模块，且两者接在不同的 SPI 总线上（如 LCD 用 SPI2、LoRa 用 SPI3），两个模块可以独立工作，互不影响。如果接在同一总线上，需要上述共享 SPI 总线处理。

如果使用同一 SPI 总线，两边的端口实现使用了相同的引脚管理模式（手动片选、轮询传输），不冲突。但互斥锁各自独立，无法跨模块保护 SPI 总线——即 LCD 和 LoRa 的总线操作可能出现交错。解决方案是在上层建立一把"SPI 总线全局锁"，两个模块在发 SPI 事务前都先获取同一把锁。

### GPIO 冲突

确保 Kconfig 或配置结构体中指定的引脚没有被其他外设（UART、I2C、PWM 等）占用，尤其是某些引脚可能有默认功能（如 JTAG 或下载器占用 GPIO 0/1/2/3）。

## 调用规范

### 基本使用流程

```c
#include "lcd12864.h"

void app_main(void)
{
    /* 1. 准备引脚配置 */
    lcd12864_config_t cfg = {
        .sclk    = 1,
        .sda     = 2,
        .rs      = 3,
        .cs      = 5,
        .reset   = 4,
        .freq_hz = 4000000,
    };

    /* 2. 创建屏幕实例（失败时返回 NULL） */
    lcd12864_ctx_t *lcd = lcd12864_create(&cfg);
    if (lcd == NULL) {
        printf("LCD init failed\n");
        return;
    }

    /* 3. 绘制画面。建议用 lock/unlock 包裹绘制序列 */
    lcd12864_lock(lcd);
    lcd12864_clear(lcd);
    lcd12864_puts(lcd, 0, 0, "Hello, LCD12864!");
    lcd12864_puts(lcd, 0, 2, "ESP32 Test");
    lcd12864_flush(lcd);
    lcd12864_unlock(lcd);

    /* 4. 不再需要时释放资源 */
    lcd12864_deinit(lcd);
}
```

### 注意事项

1. **`create` 返回 NULL 时必须处理**。可能是引脚配置错误、总线初始化失败、内存不足等原因。忽略 NULL 后续调任何函数都会崩溃。
2. **帧缓冲模式的绘制操作必须跟 `flush`**。`draw_pixel`、`fill_rect`、`putc`、`puts`、`clear` 只改内存，不直接影响屏幕。不改画面时不需要调 `flush`。
3. **多任务环境中使用 `lock`/`unlock` 配对联用**。不要在 `lock` 后忘记 `unlock`，否则其他操作屏幕的任务会永远阻塞。如果有一段绘制流程比较复杂，可以拆成多个小锁区间，避免长时间霸占锁导致其他任务饥饿。
4. **不混用光标输出和帧缓冲绘图**。`write_char`/`write_string` 使用内部光标，`putc`/`puts` 使用像素坐标系——两套 API 工作原理不同，如果混用可能导致逻辑混乱。
5. **字库只覆盖 ASCII 可打印字符**（0x20~0x7E）。汉字或特殊符号需要自行补充字库或另建字库表。非可打印字符会被统一显示为空格。
6. **`deinit` 后不能再使用句柄**。`deinit` 会释放上下文内存、帧缓冲区、互斥锁和 SPI 资源。再对同一个句柄调用任何函数都是未定义行为。

## 已知来自代码的实际情况

以下是从实际代码中能直接看出来的事实，可以帮助你更准确地理解这个模块：

- **`clear` 的实现**：`memset(handle->framebuffer, 0x00, 1024)` + 光标归零。它只清内存中的帧缓冲区，不清硬件的显示 RAM。要真的清屏必须再调一次 `flush`。
- **`write_char` 和 `putc` 都做了位转置**。它们的渲染逻辑几乎相同（8×8 位转置 + 上下半部分各 8 字节），只是参数来源不同（一个用光标坐标，一个用传入的像素坐标）。
- **端口层的 `deinit` 目前不释放 SPI 总线**（`spi_bus_free` 在函数中调用），如果其他设备还要用同一条总线，这里可能会破坏共享状态。当前实现也没有检查 `deinit` 被重复调用的情况。
- **`create` 返回 `void*`**，不是强类型的 `lcd12864_ctx_t*`。用户在使用时需要自己做一次向下转型（`(lcd12864_ctx_t *)lcd12864_create(&cfg)`）。这是因为头文件中只做了前向声明，具体类型定义在 `.c` 中，用户看不到完整结构体。
- **端口层函数不返回错误码**，都是 `void` 类型。如果 `spi_bus_initialize` 或 `spi_bus_add_device` 失败，`ESP_ERROR_CHECK` 会直接触发断言（`abort()`）。所以在正式产品固件中，可能需要将硬错误的处理方式改为返回错误码而非断言。
- **SPI 事务使用轮询模式**，所以不会产生中断占用的 CPU 开销，但也意味着在传输期间当前任务不会让出 CPU。如果屏幕刷新频率高且 SPI 时钟慢，可能会影响同等优先级任务的响应时间。
- **字库是 `static const` 数组**，编译到 .rodata 段中（只读、不占 RAM）。如果系统有 Flash 缓存压力，第一次读取字库时会触发 Cache miss，但后续读取会命中缓存。
